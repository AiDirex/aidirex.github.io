Some notes on MongoDB

I’m going to talk about MongoDB, a trending noSql solution. MongoDB is the most popular opensource NoSql databse.

NoSql

Nosql is the term used for some new emerging datastore technologies which have different data models from traditional Relational databases. There are four major types based on data model.

  • Document : MongoDB, RethinkDB, CouchDB ,…
  • Column: Hbase, Cassandra,…
  • Key-Value: redis, riak ,…
  • Graph: neo4j, …

Here I’m not going to explain each model, my focus is MongoDB. You can find several articles and books about each model over internet.

It’s about tradeoffs

When we talk about nosql solutions it always about diffrenet tradeoffs. Their are not magic boxes! Relational DBs like MySQL,PosgreSQL have gave developers usefull features like joins, transactions, ACID properties, rich query language and etc which makes developing applications a lot easier. But NoSql dbs on the other hand have taken some of this features and provided
others like easy administration, automatic sharding, automatic failover, automatic replications and some of them have been designed from beginning as a distributed system to fit in the scale out paradigm.

MongoDB

MongoDB is the most popular opensource Nosql database. Due to its popularity developers often get temped to use it without evaluating other solutions. But mongodb may not be the best choice. In this text I want to discuss about it. As I said NoSql is about tradeoffs, but does mongodb offer as much as it takes?

No joins

Mongo is a document store. It means that data is stored as semi-structured documents, in simple terms you can store data in JSON, XML or other formats. Document model is very good for simple records for e.g. if you want to store employees data they fit greatly in documents also document model is very good for one to many relations for e.g. storing employee phone numbers, but when it comes to many to many relations problems start coming e.g. keeping employee skills (people may have same skills ). There are many tutorials on how to store many to many relations in document databases. But some document stores like rethinkDB support distributed joins (they can join tables across many nodes). But MongoDB doesn’t support joins. Generally there are two solutions:

  • Doing joins in application layer.
  • Store your data like one to many relations and deal with redundancy and keeping redundancies in sync.

Storage engine

Before recent v3.2 release MongoDB’s default storage engine was MMAPv1. MMAPv1 is based on memory mapped files. A memory-mapped file is a file with data that the operating system places in memory by way of the mmap() system call. Memory mapping assigns files to a block of virtual memory with a direct byte-for-byte correlation. Memory-mapped files offer great benefits when dealing with large files but they are not optimized for real time systems. Problems start when data grows bigger than availabe memory and the OS begins to swap in and out memory pages (this problem is called Page Faults). But in the newest release, the default storage engine is WiredTiger which is optimized. so don’t forget to change your storage engine if your are on older versions.

Default Acknowledge

When you send a write query to Mongo by default it sends back an ack to client when data has been written to memory. The in memory data is synced to disk periodically. you have to configure this interval time(or generaly write concern), because there is a small window that you may loose your data.

Transactions

MongoDB doesn’t support transactions. It only has atomicity at document level. In DBs like Mysql or Posgresql you have full transactions in a single node, you only loose transactions between multiple nodes when you shard(partition) data.

Concurrency

Before v3.2 mongodb didn’t have much support for concurrency. It had database level locks which locked the whole database during writes (so no concurrent operations on the whole database!) but with the new storage engine it now supports document level locking (like row level locking in RDBMS). so upgrade to newer version sooner!

Easy ops(?)

one of the things that nosql dbs claim to have is easy administration. Routine admin tasks maybe easy doing in mongodb (they are easy every where!) but sharding your data or setting up a Highly available cluster without single point of failure is still complex manual task. but other Nosqls like rethnikDB simplify this task greatly.

Learning a new technology

One other thing you have to deal with is learning a new technology. Setting a server for testing is easy in Mongo but if you want to deploy in production you have to go throuh the docs. Also developers have to learn a new query language. because MongoDB has it’s own query language.

I’ve summarized some expriences we have faced in our project using MongoDB, finally we switched back to MySql. So before choosing MongoDB consider other solutions too.
